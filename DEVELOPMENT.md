## Patch System Overview

### Why Patches?

Ollama uses a patch-based system to maintain custom enhancements on top of the upstream llama.cpp repository. This approach:

- **Decouples ollama from llama.cpp commits**: Changes survive across upstream updates
- **Makes modifications auditable**: Each patch is a single, reviewable change
- **Enables clean vendor tree management**: Patches are applied during build, not committed to source
- **Maintains reproducibility**: Same patches always produce the same result

### Architecture

```
┌─────────────────────────────────────────────────────────┐
│ Upstream llama.cpp (ec98e2002) + Turboquant patches     │
└─────────────────────┬───────────────────────────────────┘
                      │
                      │ git clone / fetch / checkout
                      ↓
        ┌─────────────────────────────┐
        │ llama/vendor (local clone)  │
        └─────────────────────────────┘
                      │
                      │ git am (patch apply)
                      ↓
        ┌─────────────────────────────┐
        │ Patched vendor tree ready   │
        │ for building                │
        └─────────────────────────────┘
                      │
                      │ rsync
                      ↓
        ┌─────────────────────────────┐
        │ llama/llama.cpp (build src) │
        │ ml/backend/ggml/ggml (ML)   │
        └─────────────────────────────┘
                      │
                      │ CMake + CUDA
                      ↓
        ┌─────────────────────────────┐
        │ Compiled ollama binary      │
        └─────────────────────────────┘
```

### Key Files

- **`Makefile.sync`**: Defines the patch system and build workflow
  - `UPSTREAM`: GitHub URL to llama.cpp repository
  - `FETCH_HEAD`: Base commit hash (e.g., `ec98e2002`)
  - `PATCHES`: Auto-discovered from `llama/patches/*.patch`
  - Targets: `checkout`, `apply-patches`, `format-patches`

- **`llama/patches/`**: All patch files (numbered 0001-0037+)
  - Format: Unified diff (generated by `git format-patch`)
  - Named: `NNNN-descriptive-name.patch`
  - Applied in order: lower numbers first

- **`llama/vendor/`**: Local clone of upstream llama.cpp
  - Git repository (maintains history)
  - Patches applied here via `git am -3`
  - Conflict resolution available if patches fail

- **`llama/llama.cpp/`**: rsync'd copy of patched vendor (build source)
  - Read-only build artifact (generated from vendor + patches)

## Modifying the llama.cpp Backend

### Step 1: Understand Your Change

Before touching code, identify:

1. **What are you changing?**
   - New GPU kernels? New quantization types? Bug fixes? Performance improvements?

2. **Which files?**
   - CUDA: `ggml/src/ggml-cuda/*.cuh`, `ggml/src/ggml-cuda/*.cu`
   - CPU: `ggml/src/ggml-cpu/*.cpp`, `ggml/src/ggml-cpu/arch/**`
   - Common: `ggml/include/ggml.h`, `ggml/src/ggml-common.h`
   - Other: Config, types, operations, etc.

3. **Does it depend on other patches?**
   - Check existing patches to avoid conflicts
   - Example: New GPU kernels for planar3/iso3 quantization depend on turboquant being applied first

### Step 2: Create a Feature Branch in Vendor Tree

```bash
cd /home/build/ollama

# Clean the vendor tree to base commit
make -f Makefile.sync clean

# Verify patches applied
make -f Makefile.sync apply-patches

# Navigate to vendor repo
cd llama/vendor

# Create a feature branch from current HEAD (after patches applied)
git checkout -b my-feature-description

# Verify you're on the right commit
git log --oneline -1
```

### Step 3: Make Your Changes

Edit files directly in `llama/vendor/`:

```bash
# Example: Add a new GPU kernel header
vim ggml/src/ggml-cuda/my-quant.cuh

# Example: Modify flash attention
vim ggml/src/ggml-cuda/fattn-common.cuh

# Example: Register new CUDA kernels
vim ggml/src/ggml-cuda/fattn.cu
```

**Important**: 
- Changes are in `llama/vendor/` (the git repository)
- Build system will rsync these to `llama/llama.cpp/` later
- You're creating git commits that will become patches

### Step 4: Test Your Changes Locally

Build and test before formalizing as a patch:

```bash
cd /home/build/ollama

# Copy modified files to build source
# (or rsync the entire tree)
cp llama/vendor/ggml/src/ggml-cuda/my-quant.cuh llama/llama.cpp/ggml/src/ggml-cuda/

# Build
make -f Makefile.build

# Test your changes
./benchmark-direct.sh  # or your test script
```

If successful, proceed to creating the patch. If you encounter issues, fix them in vendor tree and re-test.

## Creating and Adding Patches

### Step 1: Commit Your Changes in Vendor

```bash
cd /home/build/ollama/llama/vendor

# Stage all changes
git add -A

# Commit with clear message
git commit -m "Add GPU kernels for planar3/iso3 quantization types

- Add planar-quant.cuh: Givens rotation device functions for dequantization
- Add iso-quant.cuh: Quaternion sandwich device functions for dequantization  
- Modify fattn-common.cuh: Add dequantize_V template dispatchers
- Modify fattn.cu: Register kernel instantiation for mixed KV types

Enables GPU-accelerated KV cache dequantization for RotorQuant types
alongside existing TurboQuant support. Benchmark configs: Q8_0 keys
with planar3_0 or iso3_0 values."

# Verify commit
git log --oneline -1
```

### Step 2: Generate Patch File

```bash
cd /home/build/ollama

# Generate patch from base commit (ec98e2002) to your feature branch
git -C llama/vendor format-patch \
    --no-signature \
    --no-numbered \
    --zero-commit \
    -o ../../llama/patches/ \
    ec98e2002..HEAD

# This creates: llama/patches/0001-add-gpu-kernels-for-planar3-iso3-quantization-types.patch
```

### Step 3: Number and Name the Patch

```bash
cd /home/build/ollama/llama/patches

# List existing patches to find next number
ls -1 *.patch | tail -5

# Find the next number (e.g., if 0037 is last, you need 0038)
# Rename the generated patch with leading zeros
mv 0001-add-gpu-kernels-for-planar3-iso3-quantization-types.patch \
   0038-add-rotorquant-gpu-kernels.patch

# Verify
ls -1 *.patch | tail -3
```

### Step 4: Verify the Patch Works

```bash
cd /home/build/ollama

# Clean and reset vendor tree
make -f Makefile.sync clean

# Apply all patches (yours should be last)
make -f Makefile.sync apply-patches

# Verify all patches applied successfully
ls -1 llama/patches/.*.patched | wc -l
# Should show 38 (or your patch number)

# Check for failures
tail -20 llama/patches/.*.patched 2>/dev/null || echo "No failures"
```

### Step 5: Test the Patched Build

```bash
cd /home/build/ollama

# Rebuild from scratch with new patch applied
make -f Makefile.build clean
make -f Makefile.build

# Run benchmarks or tests
./benchmark-direct.sh
```

## Building ollama

### Prerequisites

```bash
# Install Ninja (required by CMake preset)
sudo apt-get update
sudo apt-get install -y ninja-build

# CUDA toolkit (for GPU support)
# Already installed for llama.cpp CUDA 13 preset
```

### Full Build

```bash
cd /home/build/ollama

# Clean and apply patches
make -f Makefile.sync clean apply-patches

# Build with optimizations
make -f Makefile.build

# Binary location
ls -lh build_optimized/ollama
```

### Build Configuration

The build system uses a preset defined in `CMakePresets.json`:

- **Preset**: `CUDA 13`
- **Output dir**: `build_optimized/`
- **Optimization**: `-march=native -O3 -flto=thin`
- **Features**:
  - `DGGML_CUDA=ON`: NVIDIA GPU support
  - `DGGML_LTO=ON`: Link-time optimization
  - `DGGML_CUDA_GRAPHS=ON`: CUDA graph support
  - `DGGML_CUDA_FA=ON`: Flash Attention v2
  - `DGGML_CCACHE=ON`: Compiler caching

### Troubleshooting Build Issues

**"CMake was unable to find Ninja"**
```bash
sudo apt-get install -y ninja-build
```

**"Preset CMake variables error"**
- Check `CMakePresets.json` syntax
- Verify CUDA toolkit is installed: `nvcc --version`

**Compilation errors in CUDA code**
- Check include paths in `.cuh` files
- Verify all referenced device functions are declared
- Check template instantiation in `.cu` files

**Linking errors**
- Ensure all `.cu` files that use new kernels are compiled
- Check `ggml/src/ggml-cuda/CMakeLists.txt` for file registration

## Best Practices

### 1. Keep Patches Focused

**Good**: One patch = one logical change
- "Add planar3/iso3 GPU kernels"
- "Fix flash attention for new quantization types"

**Bad**: Multiple unrelated changes in one patch
- Don't combine kernel additions with performance tweaks

### 2. Write Clear Commit Messages

```
Subject: Add planar3/iso3 GPU kernel support (40-50 chars)

Body (72 char wrap):
- Add device functions for quantized dequantization
- Implement Givens rotation (planar3) and quaternion (iso3)
- Register CUDA kernels in flash attention pipeline
- Support mixed KV cache types: Q8_0 keys with new value types

Impact:
- Enables 3-bit KV cache quantization on GPU
- Expected 30-40% generation speedup vs baseline
- Requires compiled GPU support (no CPU fallback)
```

### 3. Test Before Finalizing

```bash
# Before creating patch:
1. Build successfully: make -f Makefile.build
2. Run benchmarks: ./benchmark-direct.sh
3. Check for GPU execution: grep CUDA logs
4. Verify no regressions: compare with baseline
```

### 4. Document Changes

- Add comments to complex device functions
- Update comments in modified files
- Consider adding architecture documentation
- Update this DEVELOPMENT.md if needed

### 5. Handle Conflicts

If patch application fails (`git am` conflict):

```bash
cd /home/build/ollama/llama/vendor

# Resolve conflicts in git
git diff  # see conflicts

# Edit files to fix conflicts
vim <conflicted-file>

# Continue patch application
git am --continue

# Update patch file
cd /home/build/ollama
make -f Makefile.sync format-patches
```

### 6. Versioning and Numbering

- **Patch numbers**: 0001-0037 existing, start at 0038 for new patches
- **Sequential**: Always apply in order (lower numbers first)
- **One directory**: All patches in `llama/patches/`
- **Consistent naming**: NNNN-description-with-hyphens.patch

### 7. CI/CD Considerations

When working on a team:

1. **Before submitting**: Run full test suite
2. **Patch review**: Get review of `.patch` file, not edits
3. **CI should**: 
   - Apply all patches: `make -f Makefile.sync apply-patches`
   - Build: `make -f Makefile.build`
   - Test: `./benchmark-direct.sh` or similar
4. **On merge**: Patches land in main repo unchanged

## Example Workflow: Adding RotorQuant GPU Support

This example walks through the complete process of adding planar3/iso3 GPU kernels.

### Phase 1: Setup

```bash
cd /home/build/ollama

# Ensure vendor tree is clean and patched
make -f Makefile.sync clean
make -f Makefile.sync apply-patches

# Create feature branch
cd llama/vendor
git checkout -b add-rotorquant-gpu-kernels
```

### Phase 2: Implementation

```bash
# Create new kernel files
vim ggml/src/ggml-cuda/planar-quant.cuh  # Givens rotation device functions
vim ggml/src/ggml-cuda/iso-quant.cuh     # Quaternion sandwich device functions

# Modify flash attention
vim ggml/src/ggml-cuda/fattn-common.cuh  # Add dequantize_V dispatchers
vim ggml/src/ggml-cuda/fattn.cu          # Register kernel instantiation
```

### Phase 3: Local Testing

```bash
cd /home/build/ollama

# Copy changes to build tree
cp llama/vendor/ggml/src/ggml-cuda/{planar,iso}-quant.cuh \
   llama/llama.cpp/ggml/src/ggml-cuda/

# Build
make -f Makefile.build

# Test
OLLAMA_KV_CACHE_TYPE=q8_0/planar3_0 ./benchmark-direct.sh
OLLAMA_KV_CACHE_TYPE=q8_0/iso3_0 ./benchmark-direct.sh
```

### Phase 4: Formalize as Patch

```bash
cd /home/build/ollama/llama/vendor

# Commit changes
git add -A
git commit -m "Add planar3/iso3 GPU kernel dequantization

- New: planar-quant.cuh with Givens rotation device functions
- New: iso-quant.cuh with quaternion sandwich device functions
- Modified: fattn-common.cuh to dispatch dequantization
- Modified: fattn.cu to instantiate mixed KV cache kernels"

# Generate patch
cd /home/build/ollama
git -C llama/vendor format-patch \
    --no-signature --no-numbered --zero-commit \
    -o ../../llama/patches/ \
    ec98e2002..HEAD

# Number the patch
mv llama/patches/0001-*.patch llama/patches/0038-add-rotorquant-gpu-kernels.patch
```

### Phase 5: Verify

```bash
cd /home/build/ollama

# Clean and re-apply patches
make -f Makefile.sync clean
make -f Makefile.sync apply-patches

# Should see 38 .patched files
ls llama/patches/.*.patched | wc -l

# Rebuild
make -f Makefile.build

# Final test
OLLAMA_GPU_LAYERS=999 ./benchmark-direct.sh
```

## Getting Help

- **Patch conflicts**: Check `Makefile.sync` for conflict resolution instructions
- **Build failures**: Check `cmake --build output` or run `make -f Makefile.build verbose`
- **GPU issues**: Check CUDA runtime: `nvidia-smi` and compiler: `nvcc --version`
- **Questions**: Review existing patches in `llama/patches/` for examples
